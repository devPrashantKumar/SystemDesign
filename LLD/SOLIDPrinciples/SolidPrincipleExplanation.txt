  ---
  SOLID Principles

  SOLID is an acronym for 5 design principles that make object-oriented code
  easier to maintain, extend, and understand.

  ---
  S — Single Responsibility Principle (SRP)

  ▎ A class should have only one reason to change.

  The Problem (WithoutSingleResponsibility/Invoice.java)

  public class Invoice {
      public int calculateInvoics()  { return quantity * price; }
      public void printInvoice()     { /* printing logic */ }
      public void saveToDBInvoice()  { /* DB logic */ }
  }
  This class does 3 things. If the PDF format changes, you modify Invoice. If
  the DB schema changes, you modify Invoice again. Three reasons to change =
  fragile class.

  The Fix (SingleResponsibility/)

  Invoice            → only calculates total
  PrintInvoice       → only handles printing
  InvoiceRepository  → only handles DB persistence
  Now each class has exactly one job. Changing how you print doesn't risk
  breaking the DB logic.

  ---
  O — Open/Closed Principle (OCP)

  ▎ A class should be open for extension but closed for modification.

  The Problem (WithoutOpenClosedPrinciple/AreaCalculator.java)

  public double calculateArea(Object shape) {
      if (shape instanceof Circle c)         return Math.PI * c.radius *
  c.radius;
      else if (shape instanceof Rectangle r) return r.width * r.height;
      return 0;
  }
  Every time you add a new shape (Triangle, Hexagon...), you have to modify
  AreaCalculator. This is risky — you could break existing shape logic.

  The Fix (OpenClosedPrinciple/)

  // Shape interface — defines the contract
  public interface Shape { double area(); }

  // Each shape knows its own area
  public class Circle implements Shape {
      public double area() { return Math.PI * radius * radius; }
  }

  // AreaCalculator never changes, no matter how many shapes you add
  public class AreaCalculator {
      public double calculateArea(Shape shape) { return shape.area(); }
  }
  To add a Triangle, you just create a new class — AreaCalculator is never
  touched.

  ---
  L — Liskov Substitution Principle (LSP)

  ▎ Subclasses must be substitutable for their parent class without breaking 
  ▎ behaviour.

  The Problem (WithoutLiskovSubstitutionPrinciple/)

  public class Rectangle {
      public void setWidth(int w)  { this.width = w; }
      public void setHeight(int h) { this.height = h; }
      public int area()            { return width * height; }
  }

  public class Square extends Rectangle {
      @Override
      public void setWidth(int w)  { this.width = this.height = w; } // both 
  forced equal
      @Override
      public void setHeight(int h) { this.width = this.height = h; }
  }
  If code does:
  Rectangle r = new Square();
  r.setWidth(5);
  r.setHeight(10);
  // Expected area: 50, Actual area: 100 — BROKEN
  Square silently violates Rectangle's contract. Substituting a Square where a
  Rectangle is expected breaks behavior.

  The Fix (LiskovSubstitutionPrinciple/)

  public interface Shape { int area(); }

  // Rectangle and Square are independent siblings, not parent-child
  public class Rectangle implements Shape { /* setWidth, setHeight */ }
  public class Square    implements Shape { /* setSide only */ }
  Now both can be used where a Shape is expected, and neither surprises you.

  ---
  I — Interface Segregation Principle (ISP)

  ▎ A class should not be forced to implement methods it doesn't need.

  The Problem (WithoutInterfaceSegregationPrinciple/)

  public interface Worker {
      void work();
      void eat(); // robots don't eat!
  }

  public class Robot implements Worker {
      public void work() { System.out.println("Robot is working"); }
      public void eat()  { System.out.println("Robot doesn't eat, unsupported 
  behavior"); }
  }
  Robot is forced to implement eat() — a method that makes no sense for it. This
  is a sign the interface is too fat.

  The Fix (InterfaceSegregationPrinciple/)

  public interface Workable { void work(); }
  public interface Eatable  { void eat();  }

  public class Robot implements Workable          { /* only work() */ }
  public class Human implements Workable, Eatable { /* both */ }
  Interfaces are split by what actually belongs together. Each class implements
  only what it truly supports.

  ---
  D — Dependency Inversion Principle (DIP)

  ▎ High-level modules should not depend on low-level modules. Both should 
  ▎ depend on abstractions.

  The Problem (WithoutDependecyInversionPrinciple/)

  public class DataStore {
      private MySQLDatabase db = new MySQLDatabase(); // hardcoded concrete 
  class
      public void save(String data) { db.save(data); }
  }
  DataStore (high-level) directly creates and depends on MySQLDatabase
  (low-level). If you want to switch to PostgreSQL or MongoDB, you must change
  DataStore.

  The Fix (DependecyInversionPrinciple/)

  public interface Database { void save(String data); } // abstraction

  public class DataStore {
      private Database database; // depends on interface, not MySQL
      public DataStore(Database database) { this.database = database; }
      public void save(String data) { database.save(data); }
  }

  public class MySQLDatabase implements Database { ... }
  // Tomorrow: public class MongoDatabase implements Database { ... }
  DataStore now depends on the Database interface. You can inject any
  implementation — MySQL, Mongo, in-memory for tests — without touching
  DataStore at all.
